# 3DS 强制抗锯齿插件开发计划书

## 1. 核心目标 (Core Objectives)

### 1.1 最终目标
开发一款 **Luma3DS 运行时插件**，通过**后期处理（Post-Processing）** 的方式，对新大三上屏（Top Screen）渲染的每一帧执行**形态抗锯齿（MLAA / SMAA）** 或**快速近似抗锯齿（FXAA）**，显著降低因低分辨率渲染（400x240）产生的几何边缘锯齿（Aliasing），同时尽可能将帧率损失控制在 **≤3ms/帧**。

### 1.2 性能约束
- **帧率影响**：游戏帧率下降不超过 **10%**（例如 60fps -> ≥54fps）。
- **内存开销**：额外占用 ≤ **2MB** 系统内存，**0MB** 显存（完全在 ARM11 CPU 域操作）。
- **输入延迟**：增加 ≤ **1帧** 的显示延迟（仅双缓冲排队，不允许三缓冲）。

### 1.3 兼容性目标
- 支持 **New 3DS XL** 的 804MHz CPU 模式。
- 兼容 **所有使用 GPU 渲染的游戏**（不对特定 Title ID 做硬编码）。
- 与 Luma3DS 的 **Rosalina 菜单**（亮度/滤镜/金手指）完全共存。

## 2. 工具链与平台 (Toolchain & Platform)

### 2.1 核心工具链
| 组件 | 选择 | 理由 |
| :--- | :--- | :--- |
| **编译器** | `arm-none-eabi-gcc` (devkitARM r52+) | 官方支持，可生成兼容 3DS 内核的 ABI。 |
| **构建系统** | `Makefile` (基于 devkitPro 模板) | 3DS 自制软件唯一标准，支持 `.3dsx` / `.cia` 输出。 |
| **SDK** | `libctru` (v2.4.0+) | 提供 GPU 屏幕捕获、CPU 帧缓冲访问、系统服务调用。 |
| **调试器** | `3dslink` + `gdb` (通过 Rosalina GDB stub) | 无线部署，断点调试崩溃。 |
| **性能分析** | 内嵌 `RDTSC` 计数器 (ARM11 Cycle Counter) | 微秒级测量每帧处理耗时。 |

### 2.2 目标运行环境
- **固件**：Luma3DS v13.1+ (支持 `code` 插件格式)。
- **硬件**：New 3DS XL (CPU: 804MHz, L2: 2MB, RAM: 256MB)。
- **部署路径**：`/luma/plugins/<title_id>/code.ips` (运行时替换)。

## 3. 技术路径 (Technical Roadmap)

### 3.1 阶段零：环境验证 (Day 1-2)
- **产出**：编译 `helloworld.3dsx` 并在 3DS 上通过 Rosalina 运行成功。
- **关键点**：确认 `DEVKITARM` 环境变量，以及 `3dslink` 无线传输可用。

### 3.2 阶段一：帧缓冲捕获与修改 (Day 3-5)
- **任务**：
  1. 编写 Luma3DS 插件入口，钩取 `gspWaitForVBlank` 或 `aptMainLoop`。
  2. 使用 `gfxGetFramebuffer(GFX_TOP, GFX_LEFT, NULL, NULL)` 获取上屏左眼缓冲区的原始指针。
  3. **反向验证**：将全屏像素的 R 通道置零（移除红色），观察游戏画面是否变为青色。  
     → 如果成功，证明已获得帧缓冲的写权限。
- **技术难点**：RGB565 格式的位操作（5 bits R, 6 bits G, 5 bits B）。

### 3.3 阶段二：移植轻量级抗锯齿算法 (Week 2)
- **任务**：实现一个 **FXAA 3.11 的 ARM11 优化子集**。
- **策略**：
  - 放弃高质量对比度阈值，仅实现 **边缘方向检测 + 双线性混合**。
  - 使用定点整数运算 (fixed-point) 替代浮点运算（3DS 无 FPU）。
  - 只处理 **Y 分量**（亮度）的边缘，忽略 UV。
- **预期效果**：倾斜的白色线段边缘锯齿减少 50% 以上。

### 3.4 阶段三：性能调优与双缓冲 (Week 3)
- **任务**：引入双缓冲机制，避免直接修改正在显示的前缓冲导致撕裂。
  - 从 GPU 读取当前帧到 `back_buffer`。
  - 对 `back_buffer` 执行 AA 算法。
  - 等待 VBlank，交换指针。
- **关键指标**：处理时间稳定在 **2ms 以内** (804MHz 下约 1.6M cycles)。

### 3.5 阶段四：自适应开关与游戏兼容性 (Week 4)
- **任务**：通过 Rosalina 菜单的热键（例如 `L+Up`）动态切换 AA 开关。
- **实现**：使用 `hidKeysDown()` 捕获按键，修改全局 `aa_enabled` 标志。
- **测试**：在《塞尔达传说：时之笛 3D》和《怪物猎人 XX》中运行，确认不会导致随机崩溃。

## 4. 测试实验设计 (Test Experiment Design)

### 4.1 测试环境
| 项目 | 配置 |
| :--- | :--- |
| **主机** | New 3DS XL, Luma3DS v13.1 |
| **测试游戏** | 1. 《塞尔达传说：时之笛 3D》 (边缘密集) <br> 2. 《宝可梦 究极太阳》 (色彩鲜艳) <br> 3. 《超级马里奥 3D 大陆》 (动态场景) |
| **基准工具** | 3DS 内置的 `3dbrew` 性能监视器 (Rosalina 中开启) |

### 4.2 实验一：画质改善有效性 (主观+客观)
- **方法**：
  1. 在游戏固定场景（例如林克站在房屋棱角前）拍摄关闭 AA 时的屏幕照片。
  2. 开启 AA，相同位置拍摄。
  3. **盲测**：让 5 名受试者从 10 组 A/B 对比中选出更平滑的图像。
- **通过标准**：至少 8 组被正确识别为“开启 AA 更好”，且无“过度模糊”的投诉。

### 4.3 实验二：性能开销 (客观)
- **方法**：在 `timing.c` 中嵌入如下测时逻辑：
```c
uint64_t start = getSystemTick();
perform_aa(back_buffer);
uint64_t end = getSystemTick();
uint64_t cycles = end - start;  // ARM11 核心频率约 804MHz
float microseconds = cycles / 804.0f;
```
- **数据收集**：连续运行 1000 帧，记录最大值、最小值、平均值。
- **通过标准**：平均耗时 < 2500 microseconds (2.5ms)，99th 百分位 < 4ms。

### 4.4 实验三：兼容性稳定压力测试
- **方法**：
  - 在《时之笛》中切换场景 50 次（室内、室外、战斗、菜单）。
  - 在《宝可梦》中对战 30 场，含 Z 招式特效。
- **通过标准**：0 次系统崩溃 (ARM11 data abort)，0 次画面永久花屏。

### 4.5 实验四：输入延迟量化 (可选，需要高速摄像机)
- **方法**：记录按键到画面变化间的帧数。
- **简易替代**：使用 3DS 摄像头 240fps 模式拍摄屏幕上秒表软件 + 物理按键按下瞬间。
- **通过标准**：开启 AA 后延迟增加 ≤ 1 帧 (16.67ms @60fps)。

## 5. 风险与备选方案

| 风险 | 概率 | 缓解措施 |
| :--- | :--- | :--- |
| **RGB565 转换开销过高** | 中 | 直接用 LDR 指令读取 pixel，用查表法 (LUT) 加速亮度计算。 |
| **FXAA 导致文字模糊** | 高 | 增加边缘检测的对比度阈值，避免对平坦区域 (如 UI 文本) 进行混合。 |
| **某些游戏使用自定义帧缓冲** | 低 | 降级方案：仅对 `gfxGetFramebuffer` 返回的非空有效地址执行操作。 |
| **CPU 频率波动 (268MHz 模式)** | 中 | 强制检测当前频率，若 < 800MHz，自动禁用 AA 并提示。 |
